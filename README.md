## 자체 평가
![image](https://github.com/snagryuel/projectEtoos/assets/141742542/c027bb6a-c5e2-4bdc-a54e-b729d9bc3dfd)
### 🔸 팀원 별 개인 의견
- 박상렬
  ```
  1. 역량을 정확히 알아야 할 필요성
    내 역량과 팀원의 역량을 정확히 파악하지 못해 프로젝트를 기간안에 끝내지 못함
    나와 팀원의 정확한 역량을 파악할 필요가 있음
  
  2. 진행상황의 정확한 파악
    정확한 진행상황을 파악하지 못해 현재 프로젝트의 실제 진행상황과 생각한 진행상황이 많이 달랐음
    진행상황을 정확히 파악하여 상황에 맞게 인원을 분배하는 상황 대처능력도 필요할 것으로 생각함
  
  3. 기술적인 지식의 부족
    비동기 통신, 깃, war배포등 기술적인 지식이 부족하여 해당 부분에서 프로젝트 기간을 낭비하게 되는 문제가 발생함
    해당 부분에 대한 지식을 더 익힐 필요가 있음
  ```
- 장지현
  ```
  1. 공통 모듈 정의의 부족함
    이번 프로젝트에서도 기본적으로 필요한 공통 Page, 공통 Method, Util 등을 정의를 했지만, 
    개발하면서 중간에 나오는 공통 모듈에 대해서는, 공통 모듈로 추가 정의를 하지 않고 각 메뉴별로 별도로 작성하게 되는 부분이 다소 있었음.
       ⇒ 반복적으로 작성하는 부분으로 인해 업무 퍼포먼스에 다소 지장이 생김을 느꼈고 추후 프로젝트 진행 시에는 이 경험을 바탕으로     
           프로젝트 초반 공통 모듈을 더 철저히 정의 예정
  
  2. 비동기 통신 적용의 필요성
    회원가입 아이디 중복확인, 선생님 리스트 가져오는 등 페이지 전환없이 DB에서 데이터를 가져와야 하는 부분에서 비동기 통신 기술 없이 
    개발하다 보니, 팝업으로 별도로 띄워서 개발 한다던지, 페이지 자체를 로딩 할 때 모든 데이터를 통으로 가져온다던지, 비효율적인 방법으로 
    개발하게 됨.
      ⇒ 다음 프로젝트에서는 비동기 통신 기술, 대표적으로  Ajax 기술을 꼭 적용하여 이러한 비효율적인 부분을 개선 예정
  
  3. 유동적인 역할배분의 필요성
    초반 퍼블리싱 작업을 한 명이 하다보니, Front단 진행에 속도가 붙지 않아, 프로젝트 진행 속도에 영향이 있었음.
      ⇒ 프로젝트 진행은 앞 단계부터 순차적으로 이루어져야 하는 부분을 명심하고 역할 분배를 더 유동적으로 대응할 수 있도록 개선 예정
  
  4. 업무 속도 파악의 중요성
    첫 프로젝트다 보니, 스스로 이 기능을 얼마만에 완료할 수 있는지, 서로 개발역량은 얼마나 되는지 파악에 어려움이 있었고 
    이는 결국 개발 기한을 오버하게 되는 문제로 직결되었음.
      ⇒ 스스로의 개발역량을 정확히 파악하고, 팀원의 개발역량도 최대한 파악하여 업무 진행에 영향이 없도록 개선 예정
  ```
- 장인우
  ```
  1. 기획서 요구사항 중  미완성된 기능 존재
    담당했던 Q&A 게시판을 기간 내에 완성하지 못함
    개인적으로 서블릿 라이프사이클에 대한 이해도가 많이 부족했고 경로 설정하는 작업에서 어려움이 있었음
    
  2. 기술적으로 부족했던 기능적인 부분
    회원 가입 페이지에 유효성 검사를 넣는 작업에 대해서 정규식 사용에 어려움을 겪었음
    팀원의 도움으로 인해 해결했지만 유효성 검사 부분 추가 보완할 필요성 있음 
  	 
  3. 익숙지 않았던 git의 사용방법
    git과 연동하는 부분에서 어려움을 느꼈고 full, commit, push 순서에 대해 익숙해지는 데에 시간이 걸렸음
  
  4. 업무 분담이 원활히 되지 않은 문제점
    업무 분담이 일정하게 나뉘지 않은 문제로 인해 제출 기한을 엄수하지 못했음
    개발 초기부터 담당 구역을 구분하여 실행해야할 필요성이 있었음
  ```
- 유동진
  ```
  1.처음하는 프로젝트 이다보니 예상 보다 리소스가 커짐
   일정 산출의 중요성을 다시 한 번 깨달음 
   리소스가 증가한  가장 큰  원인은 모듈을 제대로 활용하지 못해 리소스가 증가 한 것 모듈을 개발해서 활용하는 객체지향 프로그래밍의 중요성을 
   다시한 번 깨달음
  
  2.기획서 정독의 중요성
   개발 기간에 쫓기다 보니, 기획서를 제대로 읽지 못하고 개발하는 상황이 발생 . 따라서 개발해야하는 부분이 있었음에도 개발을 진행하지 못한 
   이슈가 있었고 이는 명백한 실수임. 그만큼 기획서 정독의 중요성과 커뮤니케이션이 중요하다는 사실을 다시 한 번 깨닫게 됨.
  
  3.협업의 중요성 
   프로젝트를 통해 개발은 나 혼자 하는 것이 아니라 모두가 함께 개발하는 것임을 실천으로서 알게됨. 
   내가 일정을 못지키면 모두에게 큰 피해가 돌아오게 되고 이는 팀 전체의 피해가 된 다는 것을 알게됨. 또한 팀원들의 업무 파악또한 중요하다는 
   것을 이번기회를 통해  깨달음
  
  4.공부의 중요성
   확실히 공부량이 부족해서, 내 개발능력이 뒤떨어짐을 몸소 느낌. 내가 공부를 해야. 공부한 것을 활용해야 팀에게 피해를 끼치지 않는다는 사실을 
   깨달음
  ```
